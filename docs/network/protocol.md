# 网络架构与协议设计 (Network Protocol)

Minecrust 从设计之初即为多人在线游戏（Multiplayer First），由于我们需要极致的性能以及对底层 ECS 数据的直接控制，我们**放弃**了原版 Minecraft 的 Java 协议 (Protocol 767)，转而自研一套专用的 C/S 网络架构。

## 传输层选型
为了在低延迟动作（移动、跳跃、方块破坏）和高吞吐量数据（区块下载）之间取得平衡，我们计划采用：
- **方案 A (首选)**: 基于 QUIC 协议 (如使用 Rust 的 `quinn` 库)。QUIC 基于 UDP，提供了可靠流（用于传输区块、聊天、关键事件）和不可靠数据报（用于玩家位置高频同步）的混合能力，完美契合体素游戏需求。
- **方案 B (备选)**: 纯 TCP (使用 `tokio` 异步网络)。开发简单，作为初期的脚手架方案。

## 服务器权威模型 (Authoritative Server)
所有核心逻辑都在服务器端计算并具有最终解释权：
1. **世界状态**: 服务器保存所有生成的区块和放置/破坏的方块。
2. **实体物理**: 服务器计算所有实体（包括玩家）的 AABB 碰撞、重力、击退等。
3. **验证机制**: 客户端发送“移动意图”或“破坏方块意图”，服务器验证合法性后广播给所有客户端。

## 客户端预测与平滑 (Client Prediction & Interpolation)
为了消除网络延迟带来的卡顿感，客户端需要实现：
- **输入预测 (Input Prediction)**: 客户端立即应用本地玩家的输入（移动视角、走动），而不等待服务器确认。
- **状态对齐 (Reconciliation)**: 当收到服务器发来的“权威状态”且与本地预测不符时，客户端需要平滑地修正本地位置（Rubber-banding / 回拉）。
- **实体平滑 (Interpolation)**: 收到其他玩家位置信息时，客户端需要在历史状态和最新状态之间进行线性/样条插值渲染。

## 协议包定义草案 (Packets)
数据包序列化建议采用 `bincode` 或 `prost` (Protocol Buffers)，因为我们不考虑与 Java 版兼容，所以怎么快怎么来。

### 客户端 -> 服务端 (C2S)
- `PlayerMove`: 包含 Position, Rotation (Yaw, Pitch)。
- `PlayerAction`: 开始挖掘、停止挖掘、放置方块、交互。
- `ChunkRequest`: 请求特定坐标的区块数据。

### 服务端 -> 客户端 (S2C)
- `ChunkData`: 返回 16x384x16 的压缩体素数组（采用调色板压缩）。
- `EntitySpawn`: 广播新玩家/实体出现。
- `EntityMove`: 广播其他实体的位置和朝向更新。
- `BlockUpdate`: 广播世界中某个方块的状态改变（如某人挖掉了一块泥土）。
